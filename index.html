Engineering Decision Log

Day 1: Strategy & Deep Research (April 8)

  • 4:30 PM - Initial Planning: Started analyzing the problem. The strict "not assumed, proved" constraint and the "no React/frameworks" rule felt overwhelming at first.
  • 5:15 PM - Roadmapping: To avoid getting stuck, I broke the project into 3 strict phases: 1. Deep Research (finding proof), 2. Vanilla UI Design, 3. Core JS Logic mapping.
  • 8:15 PM - Startup Deep Dive: Spent hours hunting through engineering blogs for CRED, Practo, and Swiggy. Found exactly how CRED uses Go for speed and Practo uses AWS for HIPAA compliance. Also locked in my "contrarian" setup: pitching a Django Monolith for early-stage startups instead of over-hyped microservices.

Day 2: Core Coding, Failures & Breakthroughs (April 9)

  • 10:00 AM - Prototyping: Built the barebones HTML/CSS form. Wiring up logic without React state management is a mental shift, but the core engine is successfully grabbing input values.
  • 11:30 AM - Aha Moment (The Timeline Pivot): I struggled with how to map enough unique combinations with only a few startups. Breakthrough: I don't need different companies; I need to map their lifecycle. I restructured the JS dictionary to show Zerodha's Day 1 bootstrapped stack vs. their current Enterprise stack.
  • 2:15 PM - Data Mapping: Manually connecting Swiggy's massive Python/PySpark architecture to the "Large Team / High Budget" slider logic. Tedious, but it works.
  • 5:30 PM - Failure & Recovery (DOM Rendering): Issue: Trying to render output cards dynamically using endless document.createElement lines became messy and unreadable. Fix: Scrapped it and refactored the JS to use template literals and .innerHTML. This made the code cleaner and allowed me to embed clickable proof links.
  • 7:30 PM - Failure & Recovery (Budget Logic): Issue: A simple binary "High/Low" budget was recommending ₹1 Lakh architectures to founders with only ₹50k. Fix: Rewrote the JS classification algorithm into a 3-tier system (Low: <40k, Medium: 40k-99k, High: 100k+). Now the engine prevents dangerous over-provisioning.
  • 9:00 PM - Handling Edge Cases: What happens if a user inputs 100 devs on a ₹5,000 budget? Instead of writing boring JS constraints to lock the sliders, I added "Financial Misalignment Traps" to the dictionary. The UI now actively warns the founder about runway burn and API bankruptcy.

Day 3: UI Polish & Final Deployment (April 10)

  • 8:15 AM - Upgrading Inputs: Scrapped generic dropdowns for custom <input type="range"> sliders. Wrote JS middleware to seamlessly sort raw numbers (like 45 devs) into my backend categories.
  • 9:30 AM - Color Theory: Applied the 70-20-10 UI rule. Used a warm, custom palette (#fffbf1 base, #fff2d0 surface, #e36a6a accents) to make it look like a premium SaaS product instead of a college assignment. Ensured all text is WCAG accessible.
  • 10:15 AM - Failure & Recovery (Native Dropdowns): Issue: Standard CSS couldn't override the ugly OS-level styling of native <select> dropdowns to match my new theme. Fix: Built a custom dropdown from scratch using divs and vanilla JS event listeners that silently updates a hidden <input> field. Looks premium, keeps the backend happy.
  • 11:50 AM - Final Business Logic Check: Realized a bootstrapped FoodTech startup cannot afford the Google Maps API like Swiggy. Pivoted the 'Bootstrapped + FoodTech' results away from consumer apps entirely. Mapped them to B2B SaaS models (like Petpooja POS) to prove I understand actual financial survival strategies.
  • 12:45 PM - Final Deployment: Pushed the index.html, CSS, and JS to a new GitHub repository and connected it to Vercel. I chose GitHub over a local CLI push so my commit history is public, proving my workflow. The production URL is live, the engine works flawlessly on mobile, and I have officially submitted the project.
